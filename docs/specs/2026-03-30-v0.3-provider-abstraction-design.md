# Review Council v0.3 — Provider Abstraction & New Reviewers

**Date:** 2026-03-30
**Status:** Draft
**Version:** 0.2.0 → 0.3.0

## Summary

Three changes in one release:

1. **Provider registry** — Abstraction layer for reviewer providers with auto-detection, CLI-primary/MCP-fallback transport, and graceful degradation
2. **New reviewers** — Gemini (Google) and Perplexity (Sonar API), both uniform (same prompt as existing reviewers)
3. **Skills migration** — Move from `commands/` to `skills/` directory structure for proper namespacing and ecosystem alignment

## 1. Provider Registry

### New file: `rules/providers.md`

A structured reference the orchestrator consults at runtime to detect and invoke reviewers. Each provider entry defines:

- **Name**: Human-readable label for reports
- **Detection**: How to check availability — CLI binary check (`which <cmd>`), env var, or MCP tool presence
- **CLI invocation**: Primary transport — shell command template that accepts a prompt via stdin or argument
- **MCP fallback**: Secondary transport — MCP tool name to try if CLI is unavailable
- **Env requirements**: API keys or config needed (with env var names)
- **Notes**: Provider-specific quirks (output format, token limits, etc.)

### Provider definitions at launch

#### Claude (native)
- **Detection**: Always available
- **Invocation**: `Agent` tool with `subagent_type: "reviewer-claude"`
- **MCP fallback**: None needed
- **Env requirements**: None

#### Codex (OpenAI)
- **Detection**: `which codex` succeeds OR `mcp__codex__codex` tool exists
- **CLI invocation**: `echo "$PROMPT" | codex exec --full-auto -q -`
- **MCP fallback**: `mcp__codex__codex` tool
- **Env requirements**: OpenAI API key (configured via `codex login`)

#### Gemini (Google)
- **Detection**: `which gemini` succeeds OR Gemini MCP tool exists
- **CLI invocation**: `echo "$PROMPT" | gemini -p - -o text`
- **MCP fallback**: Gemini MCP tool if configured
- **Env requirements**: Google API key (configured via `gemini login` or env)

#### Perplexity (Sonar API)
- **Detection**: `PERPLEXITY_API_KEY` env var is set
- **CLI invocation**: `curl` POST to `https://api.perplexity.ai/chat/completions` with model `sonar` and the delegation prompt as the user message
- **MCP fallback**: None (no standard MCP server)
- **Env requirements**: `PERPLEXITY_API_KEY`

### Runtime behavior

1. At the start of each review, the orchestrator probes all providers in the registry
2. Available providers are added to the reviewer pool
3. Minimum 2 providers required for convergence mode
4. If only Claude is available, run in single-reviewer mode (orchestrator provides second perspective)
5. Report header always shows what ran and what was skipped with reason:
   ```
   Reviewers: Claude, Codex, Gemini (3 of 4 — Perplexity: PERPLEXITY_API_KEY not set)
   ```

### Transport fallback logic

For each non-Claude provider:
1. Try CLI invocation
2. If CLI fails (command not found or non-zero exit), try MCP fallback
3. If both fail, skip provider and note in report
4. Never block the review because a provider is unavailable

## 2. Uniform Review Approach

All providers receive the identical delegation prompt from `rules/delegation-format.md`. No provider-specific customization of the review focus. This preserves the convergence signal: if 3/4 models independently flag the same issue, that's very high confidence.

### Round 2 handling

Currently Round 2 uses `mcp__codex__codex-reply` with a thread ID for conversational continuity. With CLI-primary transport, Round 2 sends a fresh call with the full context + Round 1 synthesis. This is better for independence — no conversational state bleeding between rounds.

The Round 2 prompt includes:
- Original context package (same as Round 1)
- Round 1 synthesis (agreed, unique, conflicting findings)
- Instructions to confirm, revise, or rebut

## 3. Skills Migration

### Directory structure change

```
# Before (commands/)
commands/
  run.md
  setup.md
  uninstall.md
agents/
  reviewer-claude.md
rules/
  orchestration.md
  delegation-format.md

# After (skills/)
skills/
  run/SKILL.md
  setup/SKILL.md
  uninstall/SKILL.md
agents/
  reviewer-claude.md
rules/
  orchestration.md
  delegation-format.md
  providers.md          (new)
```

### Why

- **Namespacing**: Claude Code auto-namespaces skills as `{plugin-name}:{skill-name}`. With `plugin.json` name `review-council` and skill name `run`, the result is `review-council:run`. No manual namespace prefixing needed.
- **Convention**: The ecosystem (superpowers, etc.) is moving to `skills/`. Following the convention means fewer surprises for users and contributors.
- **Bundling**: Skills are directories, so supporting files (templates, scripts) can live alongside the SKILL.md if needed in the future.

### Frontmatter changes

Each SKILL.md uses:
```yaml
---
name: run
description: "Multi-agent convergence review..."
---
```

The `name` field matches the directory name. The plugin system constructs `review-council:run` automatically.

### `commands/` removal

Delete the `commands/` directory entirely. No deprecation stubs needed — this is a minor version bump and the plugin isn't widely deployed yet.

## 4. Setup Command Changes

`setup` becomes lightweight. Its primary purpose shifts from "configure providers" to:

- **Verify prerequisites**: Check `gh` auth for PR reviews
- **Show provider status**: Run the same auto-detection the `run` command uses, display results
- **Manual overrides**: If a user wants to force a specific provider config (e.g., custom Codex model, Perplexity API key), guide them through setting env vars

The heavy lifting of "is this provider available?" moves into the `run` command via the provider registry.

## 5. Version Bump

0.2.0 → 0.3.0. This is a breaking change (command paths change from `commands/` to `skills/`), but the user-facing skill names remain the same (`review-council:run`, etc.).

## 6. Out of Scope (Future Work)

Ideas noted for future versions, not included in v0.3:

- **Blind spots library**: JSON files with regex triggers that inject targeted review questions
- **REVIEW.md convention**: Let repos define review focus and preferences
- **Quality scoring**: Score reviewer responses for specificity, citations, actionability
- **Scope drift detection**: Compare diff against stated intent
- **CI/PR integration**: Auto-comment review findings on PRs
- **Cost tracking**: Token usage and cost per review
- **Sonnet as distinct reviewer**: Use different Claude tiers for different perspectives
